                                                      什么是 LLM

                                                      一句话：给它一段文字，它预测接下来最可能出现的文字。不断重复，就生成了完整的回答。

                                                      输入: "The capital of France is"
输出: " Paris"

                                                      训练目标就这么简单——预测下一个 token。但当模型足够大、训练数据足够多时，它就涌现出了写代码、做推理、翻译等复杂能力。

                                                      处理流程

                                                      你的文字 → 切词(Tokenization) → 转向量(Embedding) → Transformer 处理 → 输出概率 → 选一个词 → 拼回去继续

                                                      Tokenization：把文字变成数字

                                                      模型不认字，只认数字。Tokenizer 把文字切成小块（token），每块对应一个 ID：

                                                      "我喜欢吃苹果" → ["我", "喜欢", "吃", "苹果"] → [3421, 8921, 1245, 9823]

                                                      对应用开发的意义：API 按 token 计费，prompt 有 token 上限。详见 Tokens。

                                                      Embedding：数字变成含义

                                                      每个 token ID 变成一个高维向量（比如 4096 个数字组成的数组）。语义相近的词，向量也相近：

                                                      "猫" 和 "狗" 的向量很接近
"猫" 和 "汽车" 的向量差很远

                                                      这就是为什么 RAG 能用向量搜索找到语义相关的内容。详见 Embeddings。

                                                      Transformer：理解上下文

                                                      Transformer（2017, "Attention is All You Need"）是所有主流 LLM 的基础架构。它做的核心事情是让每个词「看到」并「关注」上下文中的其他词。

                                                      Self-Attention：该看谁

                                                      "小明把书放到桌子上，因为它很重"
                              ↑
                "它" 应该关注 "书" 而非 "桌子"

                                                      每个 token 会生成三个向量来参与这个过程：

                                                      向量角色直觉
                                                      Query (Q)查询"我在找什么信息？"
                                                      Key (K)索引"我能提供什么信息？"
                                                      Value (V)内容"我实际携带的信息"

                                                      工作方式：每个 token 的 Query 和所有其他 token 的 Key 做匹配，得到相关性分数，然后按分数加权汇总所有 token 的 Value。类比 SQL：SELECT weighted_sum(V) FROM all_tokens WHERE Q matches K。

                                                      Multi-Head Attention：多个角度同时看

                                                      不是只有一组 QKV，而是有多组（多个 "head"），每组关注不同维度的关系：

                                                      "小王拿起苹果，因为他饿了"

Head 1: 他 → 小王      (指代关系)
Head 2: 饿 → 吃东西    (因果关系)
Head 3: 苹果 → 水果    (语义类别)

                                                      这就是为什么 LLM 能同时理解语法、语义、指代——不同的 head 各司其职。

                                                      Feed-Forward Network：看完之后怎么处理

                                                      Attention 把相关信息聚合过来之后，FFN 负责加工处理。研究表明模型的事实性知识主要编码在 FFN 的参数中。

                                                      类比：Attention 是 JOIN（汇聚信息），FFN 是 WHERE + 业务逻辑（加工信息）。

                                                      位置编码：告诉模型词的顺序

                                                      Transformer 本身不知道词的先后顺序（它是并行处理所有 token 的），需要额外注入位置信息。这也是为什么 context window 有长度上限——位置编码能表达的范围是有限的。

                                                      层层堆叠

                                                      一个 LLM 有几十到上百层 Transformer Block 堆叠（GPT-3 有 96 层）。浅层理解语法和局部模式，中层捕获语义关系，深层处理逻辑推理。

                                                      输出：概率采样

                                                      处理完后，模型对词表里的每个词输出一个概率：

                                                      输入: "今天天气很"

  好   65%
  热   15%
  冷    8%
  棒    5%
  ...

                                                      从中选一个（怎么选由 temperature 等参数控制），拼到输入末尾，再预测下一个。循环直到结束。

                                                      这就是为什么同样的 prompt 每次回答可能不同——因为是概率采样，不是查表。详见 Inference。

                                                      参数 = 模型的"知识"

                                                      LLM 的所有知识都编码在参数（weights）里——几十亿到几千亿个浮点数。

                                                      不是显式的规则库，而是训练出来的统计模式。类比：不是 if country == "中国": return "北京"，而是在高维空间里"中国的首都"和"北京"被拉得很近。

                                                      模型参数量
                                                      GPT-21.5B
                                                      LLaMA 38B / 70B / 405B
                                                      GPT-4 级别估计数千亿

                                                      参数越多 → 能编码的模式越复杂 → 但推理也更慢更贵。这就是为什么 API 提供商提供不同大小的模型（Haiku/Sonnet/Opus）让你在成本和能力之间取舍。

                                                      训练过程

                                                      输入: "中国的首都是"
正确答案: "北京"
模型当前预测: 上海 20% | 北京 60% | 广州 10%

→ 计算误差 → 调整参数 → 下次"北京"概率更高

                                                      在万亿 token 上重复这个过程，模型逐渐学会语言模式和世界知识。

                                                      训练完的 base model 只会「续写」，还不会「对话」。需要进一步做指令微调（SFT）和人类反馈对齐（RLHF）才能变成你用的 ChatGPT / Claude。详见 Training。

                                                      规模定律与涌现能力

                                                      Scaling Laws：为什么一直在追求更大

                                                      模型性能取决于三个因素：参数量、训练数据量、计算量。三者需要同比例扩大才能最优利用预算（Chinchilla 定律）。

                                                      这直接解释了行业趋势：不是盲目堆参数，而是用更多数据训练适当大小的模型。也解释了为什么开源社区的 7B/13B 模型越来越强——它们在用更好的数据和更长的训练来弥补参数量的不足。

                                                      涌现能力：为什么大模型突然变强

                                                      模型规模超过某个阈值后，会突然出现小模型做不到的事：

                                                      • 多步推理（Chain-of-thought）
                                                      • 代码生成与调试
                                                      • 少样本学习（给几个例子就能做新任务）

                                                      对应用开发的意义：不同任务需要不同大小的模型。简单分类/提取用小模型就够，复杂推理和创造性任务需要大模型。选模型时要基于任务复杂度做这个取舍。

                                                      对应用开发者重要的心智模型

                                                      1. LLM 是概率生成器，不是数据库 — 它可能编造事实（幻觉），因为它在做概率续写而非查表
                                                      2. 输入输出都是 token — prompt engineering 本质是在 token 空间里构造更好的上下文
                                                      3. Context window 是硬限制 — 模型只能"看到"窗口内的内容，超出就丢失
                                                      4. 模型不会学习 — 每次请求都是无状态的，上次对话的内容不会改变模型参数
                                                      5. 大模型 ≠ 一定更好 — 简单任务用小模型更快更便宜，复杂任务才需要大模型

                                                      LLM 的分类

                                                      按开放程度

                                                      类型代表选择场景
                                                      闭源 APIClaude, GPT-4, Gemini开箱即用，能力最强
                                                      开源/开放权重LLaMA, Mistral, Qwen, DeepSeek需要私有化部署或深度定制时

                                                      按能力特化

                                                      类型代表适用场景
                                                      通用对话Claude, GPT-4o大多数应用场景
                                                      推理模型Claude with extended thinking, o1/o3需要复杂逻辑推理
                                                      代码模型Claude Code, Codex编程辅助
                                                      多模态GPT-4o, Claude, Gemini需要处理图片/音频/视频
