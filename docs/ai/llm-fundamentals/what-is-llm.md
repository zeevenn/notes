## 什么是 LLM

Large Language Model（LLM，大语言模型）是一类基于大规模文本训练的生成式模型。它接收一段 token 序列，估计下一个 token 的概率分布，再把选出的 token 接回序列中继续生成。

```
输入: "The capital of France is"
输出: " Paris"
```

从训练目标看，LLM 做的是 next token prediction。这个目标本身并不复杂，但在足够大的模型规模、数据规模和计算量下，模型会学到语言结构、世界知识、代码模式以及一部分推理模式。

## 从文本到输出

```
文本 → Tokenization → Embedding → Transformer → 概率分布 → 采样 → 下一个 token
```

### Tokenization：文本离散化

模型不能直接处理原始字符，需要先由 tokenizer 把文本切成 token，并映射为整数 ID：

```
"我喜欢吃苹果" → ["我", "喜欢", "吃", "苹果"] → [3421, 8921, 1245, 9823]
```

Token 是模型计算、上下文长度和 API 计费的基本单位。不同模型的 tokenizer 不完全相同，同一段文本在不同模型下可能对应不同 token 数。详见 [Tokens](./tokens.md)。

### Embedding：从 ID 到向量

Token ID 会被查表转换为高维向量。向量本身不是“含义”的直接表示，但在训练过程中，语义和用法相近的 token 往往会形成相近的向量表示：

```
"猫" 和 "狗" 的向量很接近
"猫" 和 "汽车" 的向量差很远
```

这种表示方式也是语义搜索和 RAG 的基础。详见 [Embeddings](./embeddings.md)。

### Transformer：建模上下文

Transformer 是现代 LLM 的核心架构。它的关键能力是让序列中的每个 token 根据上下文重新计算自己的表示，而不是孤立地处理每个词。

#### Self-Attention：建立 token 之间的依赖

```
"小明把书放到桌子上，因为它很重"
                              ↑
                "它" 应该关注 "书" 而非 "桌子"
```

Self-attention 会为每个 token 生成三类向量：

| 向量      | 作用                      |
| --------- | ------------------------- |
| Query (Q) | 当前 token 要查找的信息   |
| Key (K)   | 其他 token 可被匹配的特征 |
| Value (V) | 匹配后真正参与汇总的内容  |

计算过程可以概括为：用 Query 和所有 Key 计算相关性，再按相关性加权汇总 Value。这样，每个 token 的表示都会带上与当前语境相关的信息。

#### Multi-Head Attention：并行关注不同关系

一个 attention head 只能从一个表示子空间里看上下文。Multi-head attention 会并行使用多组 QKV，让不同 head 学习不同类型的关系：

```
"小王拿起苹果，因为他饿了"

Head 1: 他 → 小王      (指代关系)
Head 2: 饿 → 吃东西    (因果关系)
Head 3: 苹果 → 水果    (语义类别)
```

这些关系不会被人工预先指定，而是在训练过程中从数据里学出来。

#### Feed-Forward Network：非线性加工

Attention 负责在 token 之间传递信息，Feed-Forward Network（FFN）负责对每个 token 的表示做进一步变换。可以把它理解为 Transformer block 中的非线性加工层。

模型的事实知识和语言模式并不存放在某个显式数据库里，而是分布在 attention、FFN、embedding 等参数中。FFN 通常被认为承载了相当一部分可被激活的模式和知识。

#### 位置编码：注入顺序信息

Self-attention 本身不包含顺序概念，因此模型需要额外的位置编码或位置嵌入来区分 token 的先后关系。位置表示方式也会影响模型能稳定处理的上下文长度。

#### 层级表示

LLM 通常由多层 Transformer block 堆叠而成。浅层更接近局部形式和语法模式，中间层逐步形成语义关系，深层则更接近任务相关的抽象表示。

### 输出：从概率分布中取样

经过最后一层后，模型会对词表中每个 token 给出一个分数，再转换成概率分布：

```
输入: "今天天气很"

  好   65%
  热   15%
  冷    8%
  棒    5%
  ...
```

解码阶段会根据采样策略选出一个 token，把它接到已有序列后面，再继续预测下一个 token。这个循环会持续到生成结束符、达到长度限制，或被外部停止。

同一个 prompt 可能得到不同回答，原因通常不是模型“改变了想法”，而是采样过程引入了随机性。详见 [Inference](./inference.md) 和 [Sampling Parameters](./sampling-parameters.md)。

## 参数、知识与能力

LLM 的参数是大量浮点数权重。训练过程通过反复调整这些权重，让模型在给定上下文时更倾向于生成合理的下一个 token。

这些参数不是显式规则库。模型内部没有类似 `if country == "中国": return "北京"` 的硬编码逻辑，而是在高维表示空间里形成了大量统计关联和可组合模式。

| 模型         | 参数量          |
| ------------ | --------------- |
| GPT-2        | 1.5B            |
| LLaMA 3      | 8B / 70B / 405B |
| 前沿闭源模型 | 通常不公开      |

参数量越大，模型通常具备更强的表达能力，但推理成本、延迟和部署复杂度也会增加。实际选型时，模型大小只是一个因素，还需要同时看数据质量、训练方式、上下文长度、工具能力和任务类型。

## 训练过程

```
输入: "中国的首都是"
正确答案: "北京"
模型当前预测: 上海 20% | 北京 60% | 广州 10%

→ 计算误差 → 调整参数 → 下次"北京"概率更高
```

预训练会在海量 token 上重复这个过程。模型不是逐条记忆所有文本，而是在压缩训练数据中的统计规律：词语搭配、事实关联、代码结构、推理范式等。

预训练后的 base model 主要擅长续写文本。要让它稳定遵循指令、保持对话格式、拒绝不安全请求，还需要经过指令微调（SFT）和偏好对齐（RLHF/DPO 等）阶段。详见 [Training](./training.md)。

## 规模定律与涌现能力

### Scaling Laws

Scaling laws 描述的是模型性能与参数量、训练数据量、计算量之间的关系。经验上，这三者需要相互匹配，单纯增加其中一个维度往往不能最优利用训练预算。

这也是近年模型训练的一个重要变化：不只是把模型做大，也会更重视数据质量、训练 token 数和后训练方法。较小模型在高质量数据和充分训练下，也可以在特定任务上取得很强效果。

### 涌现能力

当规模、数据和训练方式达到一定程度后，模型会表现出小模型难以稳定完成的能力，例如：

- 多步推理
- 代码生成与调试
- 少样本学习
- 跨语言迁移
- 工具调用和复杂指令遵循

不过，“涌现”不等于能力可靠。复杂任务仍然需要评估集、约束输出、上下文管理和错误兜底。详见 [Evaluation](./evaluation.md)。

## 应用开发中的心智模型

1. **LLM 是概率生成系统，不是事实数据库**：它可能生成看似合理但不存在的内容，需要外部知识、引用和校验机制。
2. **输入输出都受 token 约束**：prompt、检索内容、对话历史和最终回答共享同一个上下文预算。
3. **Context window 是可见范围，不是长期记忆**：模型只能使用当前请求中提供的信息；会话记忆需要由应用层维护。
4. **一次请求不会改变模型参数**：用户反馈、历史消息和工具结果只会进入上下文，不会让模型“学会”新知识。
5. **模型选择要服务任务**：分类、提取、改写等任务未必需要最大模型；复杂推理、代码迁移和长上下文任务通常更依赖强模型。

## LLM 的分类

### 按开放程度

| 类型          | 特点                         | 选择场景                     |
| ------------- | ---------------------------- | ---------------------------- |
| 闭源 API      | 能力强、接入快、细节不可控   | 快速构建产品、追求前沿能力   |
| 开源/开放权重 | 可私有化、可微调、需自行部署 | 数据敏感、成本可控、深度定制 |

### 按能力特化

| 类型     | 适用场景                         |
| -------- | -------------------------------- |
| 通用对话 | 问答、总结、改写、信息整理       |
| 推理模型 | 数学、规划、复杂决策、长链路分析 |
| 代码模型 | 代码生成、重构、调试、仓库理解   |
| 多模态   | 图像、音频、视频和文本混合输入   |
