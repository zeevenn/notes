## Evaluation

LLM 输出是概率生成结果，不是确定性程序返回值。单次 demo 只能说明模型在某个输入上表现不错，不能说明它在真实任务中可靠。上线前需要用固定评估集观察准确性、稳定性、成本和延迟。

评估主要回答这些问题：

- 这个模型能不能完成当前任务？
- 换模型、改 prompt、接入 RAG 后是否真的变好了？
- 哪些输入会失败，失败是否可接受？
- 成本和延迟是否满足产品要求？

## 评估集

评估集是一组代表真实场景的输入和期望结果。它需要覆盖常见输入、边界输入和高风险输入。

| 类型       | 示例                         | 作用                 |
| ---------- | ---------------------------- | -------------------- |
| 正常样本   | 常见问答、常规摘要、标准提取 | 验证基本能力         |
| 边界样本   | 超长输入、缺失字段、歧义问题 | 暴露鲁棒性问题       |
| 对抗样本   | prompt injection、误导性问题 | 检查安全和指令遵循   |
| 回归样本   | 历史线上失败案例             | 防止新版本重复犯错   |
| 领域样本   | 项目中的真实文档和业务术语   | 验证是否适合当前领域 |

评估集不需要一开始很大。可以从 20-50 条高质量样本开始，再把线上失败案例持续回填进去。

## 自动评测

自动评测适合结构化任务，结果可以直接比较或复现：

| 任务       | 常见指标                         |
| ---------- | -------------------------------- |
| 分类       | accuracy、precision、recall、F1  |
| 信息提取   | 字段级准确率、JSON schema 通过率 |
| RAG 问答   | answer correctness、faithfulness |
| 代码生成   | 单元测试通过率、静态检查结果     |
| 摘要       | 覆盖率、事实一致性、压缩率       |

对于自由文本，可以使用 LLM-as-judge 做辅助评审，但不要把它当成唯一标准。Judge prompt 也会有偏差，需要抽样人工复核。

## 人工评审

人工评审适合判断语气、可读性、业务正确性和复杂推理质量。评分维度要明确，避免只给“好/不好”的主观结论。

| 维度       | 判断标准                         |
| ---------- | -------------------------------- |
| 正确性     | 是否回答了问题，事实是否准确     |
| 完整性     | 是否遗漏关键步骤或限制条件       |
| 可引用性   | 是否能从上下文或来源中找到依据   |
| 指令遵循   | 是否遵守格式、语言和安全要求     |
| 简洁性     | 是否有不必要的废话或重复         |

每次修改 prompt、模型、检索策略或上下文拼装方式，都应跑同一套评估集，避免只凭主观感觉判断效果。

## 可靠性风险

### 幻觉

模型可能生成看似合理但不存在的事实。RAG、引用来源和结构化输出校验可以降低风险，但不能彻底消除。

### 上下文污染

错误信息一旦进入 context，后续生成会把它当作事实继续推理。长对话和 Agent 任务中尤其明显，需要摘要、裁剪和状态隔离。

### 分布外输入

模型在评估集之外的输入上可能表现明显下降。线上应记录失败案例，并把它们回填到回归评估集中。

### 不稳定输出

Temperature、top-p、模型版本和上下文顺序都可能影响输出。高一致性任务应使用低随机性配置，并加上格式校验。

## 上线前检查清单

- 是否有覆盖真实场景的评估集？
- 是否记录了模型、prompt、参数和检索配置？
- 是否比较了基线方案与新方案？
- 是否有失败案例和回归样本？
- 是否验证了成本、延迟和输出长度？
- 是否对结构化输出做 schema 校验？

评估的目标不是证明模型“很聪明”，而是明确它在哪些条件下可靠、在哪些条件下需要兜底。
